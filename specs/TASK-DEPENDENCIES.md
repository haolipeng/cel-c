# CEL-C 任务依赖关系详细分析

本文档详细分析每个任务的依赖关系,包括直接依赖、间接依赖和可选依赖。

---

## 📊 依赖关系分类

- **🔴 强依赖 (Blocking)**: 必须等待前置任务完成才能开始
- **🟡 弱依赖 (Soft)**: 可以并行开发,但集成时需要前置任务
- **🟢 可选依赖 (Optional)**: 前置任务完成能带来便利,但不是必须

---

## 阶段 1: 基础设施层

### Task 1.1: 项目结构与构建系统 ⚙️

**依赖关系**:
- ❌ 无依赖 (项目起点)

**被依赖关系**:
- 🔴 被 Task 1.2, 1.3, 1.4 强依赖
- 🔴 被 Task 2.1-2.6 强依赖
- 🔴 被 Task 3.1-3.4 强依赖
- 🔴 被所有后续任务间接依赖

**可并行任务**: 无 (必须最先完成)

**建议优先级**: P0 (最高,必须第一个完成)

---

### Task 1.2: 错误处理模块 (cel_error) ❌

**依赖关系**:
- 🔴 **Task 1.1**: 需要构建系统支持编译

**被依赖关系**:
- 🔴 被 Task 2.1 强依赖 (值创建需要错误处理)
- 🔴 被 Task 3.1-3.4 强依赖 (解析器错误报告)
- 🔴 被 Task 4.1 强依赖 (上下文操作错误)
- 🔴 被 Task 4.2-4.6 强依赖 (求值错误)

**可并行任务**: Task 1.3, 1.4

**建议优先级**: P0

**开发顺序**: 第 2 个开始

---

### Task 1.3: 内存管理模块 (cel_memory) 💾

**依赖关系**:
- 🔴 **Task 1.1**: 需要构建系统支持编译

**被依赖关系**:
- 🔴 被 Task 2.1 强依赖 (值的引用计数)
- 🔴 被 Task 2.5 强依赖 (AST 的 Arena 分配)
- 🟡 被 Task 4.1 弱依赖 (上下文可选使用 Arena)

**可并行任务**: Task 1.2, 1.4

**建议优先级**: P0

**开发顺序**: 第 2 个开始 (与 1.2 并行)

---

### Task 1.4: 测试框架集成 🧪

**依赖关系**:
- 🔴 **Task 1.1**: 需要构建系统配置测试目标

**被依赖关系**:
- 🟡 被所有任务弱依赖 (用于编写单元测试)

**可并行任务**: Task 1.2, 1.3

**建议优先级**: P1

**开发顺序**: 第 2 个开始 (与 1.2, 1.3 并行)

**注意**: 虽然不阻塞主线开发,但越早完成越好

**建议优先级**: P0

### Task 2.6: Optional 和 Opaque 类型 ✨

## 阶段 3: 解析器

### Task 3.1: 词法分析器 (Lexer) 🔤

**依赖关系**:
- 🔴 **Task 1.1**: 构建系统 (集成 re2c)

**被依赖关系**:
- 🔴 被 Task 3.2 强依赖 (Parser 使用 token 流)

**建议优先级**: P0

**开发顺序**: 可在阶段 2 进行时并行开发

**提示**: 初期可以使用简单的字符串表示,等 Task 2.2 完成后再集成

---

**依赖关系**:
- 🔴 **Task 3.1**: 需要词法分析器提供 token 流
- 🔴 **Task 2.5**: 需要 AST 节点定义来构建语法树

**被依赖关系**:
- 🔴 被 Task 3.3 强依赖 (宏展开需要识别宏调用)

- 🔴 **Task 3.2**: 需要 Parser 识别宏语法

- 🟡 **Task 2.4**: has() 宏需要理解 Map (可先不实现)

**被依赖关系**:
- 🔴 被 Task 3.4 强依赖 (Parser API 需要调用宏展开)
- 🔴 被 Task 4.4 强依赖 (推导式求值)

**建议优先级**: P1

**开发顺序**: Task 3.2 完成后立即开始

- 🔴 **Task 4.1**: 需要注册到上下文

- 🟡 **Task 2.2**: 字符串函数强依赖

- 🔴 依赖 Task 2.2, 2.3, 2.4 (String, List, Map)

- 🔴 依赖 Task 2.2 (String)

- 🔴 依赖 Task 2.1 (比较值)

**Task 4.5.5: Optional 函数**
- 🔴 依赖 Task 2.6 (Optional 类型)

**被依赖关系**:
- 🟡 被 Task 4.6 弱依赖 (默认上下文包含内置函数)

**可并行任务**: Task 4.2, 4.3, 4.4 (完全独立开发)

**建议优先级**: P0

**开发顺序**: 可在 Task 4.1 完成后立即开始,与 4.2-4.4 并行

**提示**: 5 个子任务可以 2-3 个工程师并行完成

## 🎯 关键路径分析 (Critical Path)

最长依赖链决定了最短完成时间:

```
1.1 (构建系统)
  → 1.2 (错误处理) + 1.3 (内存管理)
    → 2.1 (基础值类型)
      → 2.5 (AST)
        → 3.1 (词法分析器,可提前)
          → 3.2 (语法分析器)
            → 3.3 (宏展开)
              → 3.4 (Parser API)
                → 4.6 (程序 API,需要等待 4.2)
                  → 4.1 (上下文)
                    → 4.2 (求值器)
                      → 5.x (高级特性)
```

**关键路径长度**: 约 9-10 周 (理想情况)

**实际考虑并行和缓冲**: 12-16 周

## 💡 优化建议

### 1. 提前启动策略

某些任务可以在依赖未完全满足时提前开始:

- **Task 3.1 (词法分析器)**: 可在 Week 3 开始,不需要等待所有值类型
- **Task 4.5 (内置函数)**: 可以先定义接口,实现分步进行
- **Task 5.8 (文档)**: 可以边开发边写,不需要等到最后

### 2. 分阶段交付

- **Milestone 1 (Week 8)**: Task 1.1-2.5, 3.1-3.4 → Parser 完成
- **Milestone 2 (Week 12)**: Task 4.1-4.6 → MVP 完成
- **Milestone 3 (Week 16)**: Task 5.1-5.8 → 生产就绪

### 3. 风险缓冲

在关键路径上的任务预留 20% 缓冲时间:
- Task 2.1: +1 天
- Task 3.2: +1 天
- Task 4.2: +1 天
- Task 4.6: +1 天

### 4. 依赖解耦技巧

- **Mock 实现**: Task 3.1 可以先用简单字符串,后续再集成 Task 2.2
- **接口先行**: Task 4.5 可以先定义函数签名,实现分批完成
- **增量集成**: 不需要等所有依赖完成,可以逐步集成
